# 技术栈

> 技术栈不是决定性因素, 尤其是现在 AI Coding 这么盛行的时候, 但是技术栈决定了 Landing 成本

> 在 Tipsy 这样快速迭代, 强调即战力的情况下: Landing 成本是值得思考的问题

---

# 怎么判断技术栈的好坏/品味?

<div class="mt-4">

1. **主流 + 可迁移**
2. **原理 + 广度**

</div>

---

# 主流 + 可迁移

<div class="mt-4">

这里非常简单, 候选人的技术栈能够符合业务需求, 或者存在快速<span class="highlight">迁移学习</span>的可能性

</div>

<div class="mt-4">

例如: `mysql` → `PG`, `memcache` → `Redis`, `AWS` → `Aliyun` ...

</div>

---

# 原理 + 广度

<div class="mt-4">

我们需要的人才是 <span class="highlight">T 型</span>的人才, 也就是说能够有足够的技术广度帮助做好技术方案, 对某类技术有深度的研究, 能够支撑深度的思考

> 一个趋势是: 以后的工程师应该是 * 型甚至是 O 型

</div>

<div class="mt-4">

- **怎么考察原理?** 前提是作为面试官自己也要足够的清楚原理, 不然你是没有办法判断正确与否的
- **怎么考察广度?** 本质上是在实现前有没有做好充分的技术调研/Demo/横向比较优劣

</div>

---

# 实践 & 动手

> 实践能力和动手能力在 AI 时代的作用被大大的放大, 有充足的实践和犯错的经验的人是更加稀缺的

<div class="mt-4">

1. **Github 项目**
2. **业余商业项目**
3. **对某项技术的复现/Demo**
4. **除了专业技术之外:**
   - 独立运营产品
   - UI/UX Design
   - 遥测产品的数据

</div>

# AI Coding

<div class="mt-2">

1. **有无 AI 产品的订阅**: 如果没有基本可以直接淘汰掉了
2. **有无 AI Coding 的订阅**: 如果没有基本可以直接淘汰掉了
3. **常用的 AI 产品有什么**: 除了 ChatGPT / Gemini 之外
4. **Agents.md / Skills / MCP** 的原理/使用
5. **Subagent / Agent Teams / Context Engineering / Harness Engineering / Programmatic tool calling**

</div>

<div class="mt-4">

> 简历上写的垃圾 RAG / 企业客服直接过滤掉, 没有任何价值

直接问: <span class="highlight">embedding 的维数</span> + <span class="highlight">多模态数据的 embedding 方案</span>

</div>

# 总结

<div class="mt-4">

1. **多问为什么**
2. **广度和深度同样重要**

4. <span class="warning">质疑 AI Native</span>: 宁愿要一个真正的前端/后端/QA, 也不想要一个虚假的标榜 AI Native 工程师
